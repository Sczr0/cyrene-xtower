## 翻译概览

Cyrene 项目支持三种语言：
- 英语 (en) - 基础语言，所有翻译的基准
- 简体中文 (zh-cn) - 默认语言
- 繁体中文 (zh-tw) - 部分翻译，使用简体中文作为回退

作为翻译人员，您需要为特定语言提供准确、自然的翻译。

## 翻译工具和文件

### 翻译文件

项目的翻译主要分布在两个地方：

1. **Inlang 消息文件** - 位于 `messages/` 目录下的 JSON 文件，适用于简单文本和模板消息
   - `messages/en.json` - 英语（基础语言）
   - `messages/zh-cn.json` - 简体中文
   - `messages/zh-tw.json` - 繁体中文

2. **TypeScript 本地化文件** - 位于 `src/lib/i18n/locales.ts`，包含结构化的复杂文本

### 翻译格式

#### JSON 消息格式

简单的翻译使用 JSON 格式：

```json
{
  "$schema": "https://inlang.com/schema/inlang-message-format",
  "hello_world": "Hello, {name} from en!",
  "button_submit": "Submit"
}
```

#### 模板参数

翻译中可以包含参数，使用 `{参数名}` 表示：

```json
{
  "welcome_message": "Welcome, {name}!"
}
```

#### 复数形式

使用 Inlang 的复数语法表示不同的数量：

```json
{
  "item_count": "{count, plural, one {# item} other {# items}}"
}
```

## 翻译流程

1. **确认翻译任务** - 从项目维护者处获取需要翻译的内容
2. **下载最新文件** - 获取当前版本的翻译文件
3. **进行翻译** - 按照规范翻译文本
4. **质量检查** - 检查翻译的准确性和一致性
5. **提交翻译** - 通过 PR 或邮件提交翻译文件
6. **反馈修改** - 根据反馈进行必要的修改

## 翻译规范

### 语言一致性

1. **保持术语一致性** - 使用相同的词汇翻译相同的概念
2. **保持风格一致性** - 保持整个项目的翻译风格统一
3. **遵守语法规则** - 遵守目标语言的语法规则

### 数字和单位

1. **数字格式** - 使用目标语言习惯的数字格式
2. **货币** - 使用对应语言的货币表示
3. **百分号** - 保持与源语言一致的位置

### 特殊注意事项

1. **占位符** - 不要翻译 `{name}` 这样的占位符
2. **HTML 标签** - 不要修改或翻译 HTML 标签
3. **特殊字符** - 保持特殊字符不变
4. **大小写** - 遵守目标语言的大小写规则

## 质量检查

### 准确性检查

1. 翻译是否准确表达了原文的意思
2. 是否有遗漏或误译的内容
3. 专业术语是否使用正确

### 语言质量检查

1. 语法是否正确
2. 语言是否自然流畅
3. 标点符号使用是否正确
4. 拼写是否正确

### 一致性检查

1. 相同术语翻译是否一致
2. 风格是否统一
3. 格式是否正确

## 提交翻译

### 通过 Pull Request

1. Fork 项目到您的 GitHub 账户
2. 创建新的分支进行翻译工作
3. 完成后提交 Pull Request
4. 在 PR 描述中说明您所做的更改

### 邮件提交

如果您不是开发者，也可以通过邮件提交翻译：

1. 将翻译文件附在邮件中
2. 说明您所做的更改
3. 发送至项目维护者的邮箱

## 常见问题

### Q: 如何处理文化差异？

A: 在保持原意的基础上，适当调整表达方式以适应目标语言的文化习惯。例如，对于一些笑话或习语，可能需要意译而不是直译。

### Q: 如何处理专有名词？

A: 对于专有名词，如果官方已有翻译，优先使用官方翻译。如果没有，可音译或意译，并在术语表中注明。

### Q: 翻译文件中出现 `null` 值怎么办？

A: 这表示该项没有翻译，需要您提供翻译。不要保留 `null` 值。

### Q: 如何检查翻译的完整性？

A: 可以使用 diff 工具比较您的翻译文件和基础语言文件，确保所有键都有对应的翻译。

### Q: 如何处理过长的文本？

A: 如果翻译文本过长，需要考虑布局和显示问题。可以适当精简表达，但不要遗漏重要信息。
